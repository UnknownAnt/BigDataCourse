# 六、 AI 协作思考题

**题目**：在顺利完成本周的并发改造后，请对"异步并发 vs 同步串行"进行深刻的工程反思。

### (1) 从"能跑通"到"跑得快"——这一周我学到了什么

上周做实验九的时候，跑通 5 条数据用了 14 秒，我当时觉得"还行啊，不到 3 秒一条，也没多慢"。然后我在思考题里算了一下 100 万条要 33 天，把自己吓了一跳。但说实话，那时候我对"33 天"这个数字的感知还是抽象的——毕竟只是算术题，不是真的跑过。

这周我亲手跑了 1000 条数据，才算真正体会到了差距。同步串行模式下，1000 条理论上要 48 分钟。而改成异步并发之后，实际只用了不到 2 分钟。这个对比是实实在在的，不是纸上算出来的。当我看到 tqdm 进度条"嗖嗖嗖"地往前冲的时候，我第一次对"并发"这两个字有了体感。

说白了，上周我是在"想"并发有多快，这周我是在"感受"并发有多快。这两者完全是不一样的。

### (2) @retry 装饰器——一个让我改变认知的设计

做实验九的时候，我的 `extract_features()` 函数里有一个 `try-except` 块，遇到错误就返回一个"API错误"字典，然后继续跑下一条。当时我觉得这就算"容错"了——程序不会崩嘛，够了。

但这周我意识到，这种"遇到错误就跳过"的策略其实是在默默吞掉数据。如果 API 偶尔超时，我跳过了，那我的结果里就少了一条数据，而且我可能根本没注意到。1000 条里跳过了 3 条，看起来"成功率 99.7%"很好看，但那 3 条可能是非常关键的数据。

`@retry` 装饰器的思路完全不同：它不是"失败就算了"，而是"失败了就再来一次，而且越等越久再试"。这个设计背后其实有一个很朴素的道理——很多错误是暂时的。网络抖动、API 限流、服务器重启，这些都是几秒钟就能恢复的问题。如果你第一次失败就放弃，那就太草率了。但如果你立刻疯狂重试，又可能把人家服务器打崩。所以"等一会儿再试，而且越失败等越久"，是一个非常优雅的平衡。

我在代码里特意加了一段逻辑：只对 429（限流）和 timeout（超时）这类"临时性"错误进行重试，对于 JSON 解析错误这种"永久性"错误直接返回兜底字典。这个区分很重要——你不会因为 JSON 格式错了就重试，因为再试几次结果还是一样的。

### (3) Semaphore——"慢就是快"的哲学

并发度设为 20，而不是 100 或 1000，这个选择让我想了很久。理论上并发越高越快，为什么不直接开 500 个协程？

答案很简单：API 平台会限流。我第一次测试的时候手滑把并发设成了 100，结果大量请求收到 429 错误，触发了 @retry 的指数退避，整体反而比并发 20 慢了很多。这就像节假日的高速公路——车太多，大家都堵在路上，还不如限流放行来得快。

`Semaphore(20)` 的作用就是控制"同时在飞的请求数"不超过 20。其余的请求在 `await` 处排队，等前面的完成了再上去。这看起来像是"限制了速度"，但实际上是在保护系统——不让 API 平台限流，不让网络带宽被打满，不让重试机制被触发。有时候"慢一点"反而"快一点"，这大概就是工程里的"欲速则不达"吧。

### (4) 从脚本到系统——我理解的"工程化"

做完实验九和实验十，我最大的感悟是：**能跑通 5 条数据的脚本和能稳定处理 1000 条数据的系统，之间的距离远比我想的大。**

上周写同步版本的时候，我只需要关注"逻辑对不对"——Prompt 设计好了、JSON 能解析了、CSV 能保存了，就 OK 了。但这周做异步版本，我需要额外考虑：

- **并发度设多少？** 太高会被限流，太低浪费性能。需要试错找到平衡点。
- **重试几次？** 等多久？等待策略是线性增长还是指数增长？
- **怎么区分临时错误和永久错误？** 429 要重试，JSON 解析错误不用。
- **进度怎么展示？** 1000 条跑 2 分钟，如果没有任何输出，用户会以为程序卡死了。
- **结果怎么验证？** 1000 条里有多少成功、多少失败？失败的那些是什么原因？

这些问题在写小脚本的时候根本不会想到。但一旦数据量上来、对稳定性有要求，它们就全都冒出来了。我终于理解了老师在课堂上说的那句话："写代码和写系统是两件完全不同的事。"
