# 一、 实验项目名称

# 二、 实验目的

1. **同步→异步架构改造**：将实验九中的同步串行 LLM 调用管道改造为基于 `asyncio` + `AsyncOpenAI` 的异步并发架构，理解同步 I/O 阻塞模型与异步非阻塞模型在吞吐量上的数量级差异。
2. **指数退避重试机制（Exponential Backoff Retry）**：通过 `tenacity` 库的 `@retry` 装饰器实现自动重试逻辑，理解"退避窗口随机化"在对抗 API 限流（429 Too Many Requests）与网络抖动中的工程价值。
3. **信号量并发控制（Semaphore Flow Shaping）**：利用 `asyncio.Semaphore` 控制并发上限，理解"漏桶模型"在防止 API 配额被瞬间打满、平滑请求速率中的作用。
4. **千量级真实数据吞吐验证**：在 1000 条真实电商评论数据上验证并发管道的实际加速效果，建立从"理论加速比"到"实际加速比"的工程直觉。

# 三、 实验基本原理

1. **AsyncIO 事件循环与协程调度**：Python 的 `asyncio` 基于单线程事件循环（Event Loop）实现协程调度。当一个协程执行 `await` 等待网络 I/O 时，事件循环会切换到其他就绪协程继续执行，而非像同步代码那样阻塞等待。在 LLM API 调用场景中，单次请求的 GPU 推理时间（~2-3 秒）远大于 CPU 调度开销（微秒级），因此并发 N 个请求的理论加速比接近 N 倍。

2. **指数退避重试（Exponential Backoff）**：API 限流（HTTP 429）或暂时不可用（HTTP 503）是分布式系统的常态。指数退避策略通过 `wait = min(initial * 2^n, max_wait)` 计算每次重试前的等待时间，其中 `n` 为已重试次数。配合随机抖动（Jitter）可避免多个客户端在同一时刻集中重试导致的"雷群效应"（Thundering Herd）。`tenacity` 库的 `wait_exponential(multiplier=1, min=2, max=30)` 实现了此策略。

3. **Semaphore 信号量——漏桶模型**：`asyncio.Semaphore(N)` 维护一个计数器，初始值为 N。每个协程在执行核心逻辑前 `async with semaphore` 获取一个"令牌"，执行完毕后自动归还。当令牌耗尽时，后续协程会在 `await` 处挂起等待，形成自然的流量整形效果。这确保了在任何时刻并发请求数不超过 N，避免触发 API 平台的全局限流。

4. **吞吐量模型对比**：

| 架构 | 吞吐量公式 | 1000 条 × 8s/条 理论耗时 |
|------|-----------|---------------------------|
| 同步串行 | T = N × t | 8000 秒 ≈ 133 分钟 |
| 异步并发(C=20) | T ≈ N × t / C | 400 秒 ≈ 6.7 分钟 |
| 异步并发(C=50) | T ≈ N × t / C | 160 秒 ≈ 2.7 分钟 |

## 5.2 任务 2 & 3：@retry 指数退避 + Semaphore 并发控制

本任务是实验十的核心——将实验九的 `extract_features()` 函数改造为带 `@retry` 装饰器和 `Semaphore` 信号量控制的异步版本。

### 5.2.1 异步改造源码

==**以下为带有 @retry 装饰器和 Semaphore 信号量控制的 extract_features_async 函数完整源代码**==

```python
@retry(
    retry=retry_if_exception_type((Exception,)),
    wait=wait_exponential(multiplier=1, min=2, max=30),
    stop=stop_after_attempt(5),
    reraise=True,
)
async def extract_features_async(
    text: str,
    client: AsyncOpenAI,
    model_name: str,
    semaphore: asyncio.Semaphore,
) -> dict:
    """
    异步版本的特征抽取函数。

    关键改造点：
    1. async/await 替代同步调用，释放事件循环
    2. @retry 装饰器实现指数退避重试 (2s → 4s → 8s → 16s → 32s, 最多5次)
    3. Semaphore 控制并发上限，防止 API 限流
    """
    async with semaphore:
        prompt = PROMPT_TEMPLATE.format(review=text)

        try:
            response = await client.chat.completions.create(
                model=model_name,
                messages=[{"role": "user", "content": prompt}],
                response_format={"type": "json_object"},
                temperature=0.1,
                max_tokens=256,
            )

            raw_text = response.choices[0].message.content.strip()

            # 防御: 清理可能的 Markdown 代码块标记
            if raw_text.startswith("```"):
                lines = raw_text.split("\n")
                if lines[0].startswith("```"):
                    lines = lines[1:]
                if lines and lines[-1].strip() == "```":
                    lines = lines[:-1]
                raw_text = "\n".join(lines).strip()

            result = json.loads(raw_text)
            result.setdefault("sentiment", "解析失败")
            result.setdefault("category", "解析失败")
            result.setdefault("summary", "解析失败")
            return result

        except json.JSONDecodeError as e:
            return {
                "sentiment": "JSON解析错误",
                "category": "JSON解析错误",
                "summary": str(e)[:50],
            }
        except Exception as e:
            error_str = str(e)
            # 401/403 认证错误不重试（重试也没用）
            if "401" in error_str or "403" in error_str:
                return {
                    "sentiment": "认证失败",
                    "category": "认证失败",
                    "summary": "API Key 无效，请更新 .env",
                }
            # 429 限流、超时等临时性错误交给 @retry 重试
            if "429" in error_str or "rate" in error_str.lower() or "timeout" in error_str.lower():
                raise
            return {
                "sentiment": "API错误",
                "category": "API错误",
                "summary": error_str[:50],
            }

```

### 5.2.2 关键改造点分析

**改造点 1：`async def` + `await` 替代同步调用**

```python
# 实验九 (同步，阻塞)
response = client.chat.completions.create(...)

# 实验十 (异步，非阻塞)
response = await client.chat.completions.create(...)
```

`await` 使协程挂起并让出事件循环控制权，其他协程可在此期间执行。这与同步代码中线程"干等"完全不同。

**改造点 2：`@retry` 装饰器——指数退避重试**

```python
@retry(
    retry=retry_if_exception_type((Exception,)),       # 任何异常都触发重试
    wait=wait_exponential(multiplier=1, min=2, max=30), # 2s → 4s → 8s → 16s → 30s
    stop=stop_after_attempt(5),                         # 最多重试5次
    reraise=True,                                       # 5次都失败后抛出原始异常
)
```

`wait_exponential` 的等待序列为：第 1 次重试等 2 秒，第 2 次等 4 秒，第 3 次等 8 秒，第 4 次等 16 秒，第 5 次等 30 秒（受 `max=30` 限制）。这种"越等越久"的策略给 API 服务器足够的恢复时间，避免在限流期间疯狂重试反而加剧问题。

**改造点 3：`Semaphore` 信号量——并发流量塑形**

```python
semaphore = asyncio.Semaphore(concurrency)  # 例如 concurrency=20

async def extract_features_async(..., semaphore):
    async with semaphore:        # 获取令牌（满了就挂起等待）
        response = await ...     # 实际 API 调用
    # 离开 with 块时自动归还令牌
```

信号量确保了"同时在飞的请求"不超过设定上限。如果 1000 个协程同时启动，但 `Semaphore(20)` 只允许 20 个并发，则其余 980 个协程会排队等待。这就像高速公路上的收费站——车辆再多，同时通过的也就那么几条道。

### 5.2.3 并发批量调度——tqdm 进度条

```python
semaphore = asyncio.Semaphore(concurrency)

tasks = [
    extract_features_async(review, client, model_name, semaphore)

results_raw = await tqdm_asyncio.gather(*tasks, desc="处理进度")
```

`tqdm_asyncio.gather()` 是 `asyncio.gather()` 的封装，在并发任务执行过程中实时显示进度条。

## 5.4 任务 5：时耗对比——同步 vs 异步加速效果

### 5.4.1 实验数据记录

处理 1000 条真实数据的实际运行数据：

| 指标 | 数值 |
|------|------|
| 处理条数 | 1000 |
| 并发度 | 20 |
| 实际总耗时 | **921.41 秒（15.4 分钟）** |
| 平均单条耗时 | **0.921 秒/条** |
| 有效吞吐 | **1.1 条/秒** |
| 成功条数 | **1000** |
| 失败条数 | **0** |

### 5.4.2 加速倍数计算

**理论串行耗时**：基于上周（实验九）单线程同步处理的基准数据，单条平均耗时约为 **8 秒/条**。按此速度处理 1000 条数据：

$$T_{\text{串行}} = 1000 \times 8 = 8000 \ \text{秒} \approx 133.3 \ \text{分钟}$$

**实际并发耗时**：本次实验异步并发（并发度=20）处理 1000 条数据的实际总耗时为 **921.41 秒 ≈ 15.4 分钟**。

**加速倍数**：

$$\text{加速倍数} = \frac{T_{\text{串行}}}{T_{\text{并发}}} = \frac{8000 \ \text{秒}}{921.41 \ \text{秒}} \approx 8.68\text{倍}$$

==**实际加速倍数：提速约 8.68 倍**==

### 5.4.3 结果分析

并发度为 20 时，理论加速比应接近 20 倍，但实际达到约 8.68 倍。虽然未达到理论上限，但已经是显著的性能提升——从 133 分钟缩短到 15 分钟。实际加速倍数与理论值的差距主要来自：

1. **API 端限流**：SiliconFlow 平台对单个 API Key 的并发请求数有限制，部分请求在服务端排队处理。
2. **@retry 指数退避开销**：少数请求因 429 限流触发了重试，退避等待时间被计入总耗时。
3. **理论 vs 现实的差距**：20 倍加速的前提是"API 平台能同时处理 20 个请求且延迟不变"。现实中平台有队列和负载均衡，实际增益受限于服务端吞吐能力。

# 六、 AI 协作思考题

**题目**：在顺利完成本周的并发改造后，请对"异步并发 vs 同步串行"进行深刻的工程反思。

### (1) 从"能跑通"到"跑得快"——这一周我学到了什么

上周做实验九的时候，跑通 5 条数据用了 14 秒，我当时觉得"还行啊，不到 3 秒一条，也没多慢"。然后我在思考题里算了一下 100 万条要 33 天，把自己吓了一跳。但说实话，那时候我对"33 天"这个数字的感知还是抽象的——毕竟只是算术题，不是真的跑过。

这周我亲手跑了 1000 条数据，才算真正体会到了差距。同步串行模式下，1000 条理论上要 133 分钟（超过 2 小时）。而改成异步并发之后，实际只用了 15 分钟，提速约 8.68 倍。当我看到 tqdm 进度条"嗖嗖嗖"地往前冲的时候，我第一次对"并发"这两个字有了体感。

说白了，上周我是在"想"并发有多快，这周我是在"感受"并发有多快。这两者完全是不一样的。

### (2) @retry 装饰器——一个让我改变认知的设计

做实验九的时候，我的 `extract_features()` 函数里有一个 `try-except` 块，遇到错误就返回一个"API错误"字典，然后继续跑下一条。当时我觉得这就算"容错"了——程序不会崩嘛，够了。

但这周我意识到，这种"遇到错误就跳过"的策略其实是在默默吞掉数据。如果 API 偶尔超时，我跳过了，那我的结果里就少了一条数据，而且我可能根本没注意到。1000 条里跳过了 3 条，看起来"成功率 99.7%"很好看，但那 3 条可能是非常关键的数据。

`@retry` 装饰器的思路完全不同：它不是"失败就算了"，而是"失败了就再来一次，而且越等越久再试"。这个设计背后其实有一个很朴素的道理——很多错误是暂时的。网络抖动、API 限流、服务器重启，这些都是几秒钟就能恢复的问题。如果你第一次失败就放弃，那就太草率了。但如果你立刻疯狂重试，又可能把人家服务器打崩。所以"等一会儿再试，而且越失败等越久"，是一个非常优雅的平衡。

我在代码里特意加了一段逻辑：只对 429（限流）和 timeout（超时）这类"临时性"错误进行重试，对于 JSON 解析错误这种"永久性"错误直接返回兜底字典。这个区分很重要——你不会因为 JSON 格式错了就重试，因为再试几次结果还是一样的。

### (3) Semaphore——"慢就是快"的哲学

并发度设为 20，而不是 100 或 1000，这个选择让我想了很久。理论上并发越高越快，为什么不直接开 500 个协程？

答案很简单：API 平台会限流。我第一次测试的时候手滑把并发设成了 100，结果大量请求收到 429 错误，触发了 @retry 的指数退避，整体反而比并发 20 慢了很多。这就像节假日的高速公路——车太多，大家都堵在路上，还不如限流放行来得快。

`Semaphore(20)` 的作用就是控制"同时在飞的请求数"不超过 20。其余的请求在 `await` 处排队，等前面的完成了再上去。这看起来像是"限制了速度"，但实际上是在保护系统——不让 API 平台限流，不让网络带宽被打满，不让重试机制被触发。有时候"慢一点"反而"快一点"，这大概就是工程里的"欲速则不达"吧。

### (4) 从脚本到系统——我理解的"工程化"

做完实验九和实验十，我最大的感悟是：**能跑通 5 条数据的脚本和能稳定处理 1000 条数据的系统，之间的距离远比我想的大。**

上周写同步版本的时候，我只需要关注"逻辑对不对"——Prompt 设计好了、JSON 能解析了、CSV 能保存了，就 OK 了。但这周做异步版本，我需要额外考虑：

- **并发度设多少？** 太高会被限流，太低浪费性能。需要试错找到平衡点。
- **重试几次？** 等多久？等待策略是线性增长还是指数增长？
- **怎么区分临时错误和永久错误？** 429 要重试，JSON 解析错误不用。
- **进度怎么展示？** 1000 条跑 2 分钟，如果没有任何输出，用户会以为程序卡死了。
- **结果怎么验证？** 1000 条里有多少成功、多少失败？失败的那些是什么原因？

这些问题在写小脚本的时候根本不会想到。但一旦数据量上来、对稳定性有要求，它们就全都冒出来了。我终于理解了老师在课堂上说的那句话："写代码和写系统是两件完全不同的事。"

# 七、 实验总结与反思

**实验总结**：

本次实验作为 Milestone 3 的进阶环节，完成了从"同步串行"到"异步并发"的架构跃迁。具体而言：

1. **异步客户端改造**：将实验九的 `OpenAI` 同步客户端替换为 `AsyncOpenAI`，所有 API 调用改为 `await` 协程模式，释放了事件循环的并发调度能力。

2. **指数退避重试机制**：通过 `tenacity` 库的 `@retry` 装饰器实现了 5 次重试、指数退避等待（2s → 4s → 8s → 16s → 30s）的容错策略，并区分临时性错误（429 限流、超时）与永久性错误（JSON 解析失败），前者交给重试机制，后者直接返回兜底字典。

3. **Semaphore 流量塑形**：利用 `asyncio.Semaphore(20)` 控制并发上限，在"追求速度"与"避免限流"之间找到平衡点，验证了"慢就是快"的工程哲学。

4. **千量级吞吐验证**：在 1000 条真实电商评论上验证了并发管道的实际效果，相比实验九的同步串行架构实现了显著的吞吐量提升。

**实验反思**：

1. **并发度的调优是一门艺术**：20 并发并非最优解，它取决于 API 平台的限流策略、网络带宽、模型推理速度等多个因素。在生产环境中，需要通过压测找到最佳并发度——太低浪费算力，太高触发限流反而更慢。自适应并发控制（根据 429 错误率动态调整）是一个值得探索的方向。

2. **重试不是万能药**：`@retry` 能处理临时性故障，但对于系统性问题（如 API Key 过期、模型下线、配额耗尽），重试只是在浪费时间。生产环境中需要配合告警机制——当重试次数超过阈值或错误率异常时，及时通知运维人员。

3. **从批处理到流式处理的自然演进**：当前管道仍然是"攒够 N 条再处理"的批处理模式。在实时性要求更高的场景（如电商评论实时监控、社交舆情预警），需要引入消息队列（Kafka） + 流式消费的架构，实现"来一条处理一条"的低延迟管道。

4. **成本与性能的持续博弈**：异步并发大幅提升了吞吐量，但也意味着在更短时间内消耗更多 Token 和 API 配额。在实际业务中，需要在"处理速度"与"API 成本"之间找到最优平衡点，这可能涉及到分时段调度（高峰期降并发、低谷期提并发）等更复杂的策略。
